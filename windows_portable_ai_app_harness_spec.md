# Windows Portable AI App Harness Specification
Version: 1.0

---

# Purpose

This document defines the required architecture, technology stack, packaging strategy, deployment expectations, and development guardrails for all generated applications.

The goal is to ensure every generated application:

- Runs on Windows 11
- Requires no administrator privileges
- Requires no manual installation of Python, Node.js, or other runtimes
- Can be launched by a non-technical user via double-click
- Is portable and easy to distribute internally
- Uses a modern browser-style UI
- Supports local data storage and Excel import/export
- Can be reliably packaged and shared between team members

The AI coding agent MUST follow this specification unless explicitly overridden.

---

# Core Philosophy

Applications generated under this specification are intended to behave like modern portable Windows desktop applications while internally using web technologies.

The recipient of the application should:

1. Receive a ZIP file
2. Extract the ZIP file
3. Double-click a batch file
4. Immediately use the application

The recipient should NEVER need to:

- Install Python
- Install Node.js
- Open a terminal
- Run CLI commands
- Configure environment variables
- Install Docker
- Install package managers
- Use administrator privileges

All required runtimes and dependencies MUST be bundled with the application.

# Portable Deployment Requirements

Applications must be:

- portable
- self-contained
- runnable from any folder
- easy to distribute internally

The preferred deployment model is:

```text
AppName.zip
```

Containing:

```text
AppName/
├── Run_App.bat              User-facing launcher — double-click to start the app
├── AppName.exe              PyInstaller-generated executable — bootstraps the Python runtime
└── _internal/               Generated by PyInstaller — contains everything the app needs to run
    ├── python3xx.dll        Bundled Python runtime (no system Python required)
    ├── (pip packages)       All Python dependencies bundled at package time
    ├── (frontend assets)    Compiled React build output (HTML, CSS, JS) — served by FastAPI
    ├── (app icons/images)   Static assets from the project assets/ folder
    └── (DLLs)               SQLite, MSVC runtime, and other required native libraries
```

The user experience should be:

1. Extract ZIP
2. Double-click `Run_App.bat`
3. Application launches

### Frontend Dependency Pinning

All frontend dependencies MUST be pinned to exact versions in `package.json`.

Use exact version pinning:

```json
{
  "dependencies": {
    "react": "18.3.1",
    "recharts": "2.12.7"
  }
}
```

Do NOT use range specifiers:

```json
{
  "dependencies": {
    "react": "^18.0.0",   ← forbidden
    "recharts": "~2.12.0" ← forbidden
  }
}
```

A `package-lock.json` MUST be committed alongside `package.json`.

# Local Storage Requirements

Applications MUST store user data under:

```text
%LOCALAPPDATA%/AppName
```

Where `AppName` is the name of the specific application being built.

This includes:

- SQLite databases
- exports
- logs
- cache files
- temp files
- user configuration

Applications MUST NOT write user data into:

- installation folder
- Program Files
- Windows system directories

Applications MUST automatically create this directory at startup if it does not exist. The launcher MUST call the equivalent of:

```python
os.makedirs(app_data_dir, exist_ok=True)
```

This MUST happen before any database access, file writes, or logging is attempted.

## PyInstaller Configuration Requirements

The PyInstaller `.spec` file MUST include:

- `console=False` (equivalent to `--windowed`) to suppress the console window
- `noconsole=True` where applicable

Without this, a terminal window will appear behind the application window on launch.

Example `.spec` entry:

```python
exe = EXE(
    ...
    console=False,
    ...
)
```

---

## MSVC Runtime DLL Requirements

PyInstaller does not always bundle the Microsoft Visual C++ runtime DLLs automatically. On stripped-down or corporate-hardened Windows machines these DLLs may be absent, causing the application to crash immediately before Python starts — with no useful error message shown to the user.

The PyInstaller `.spec` file MUST explicitly include the MSVC runtime DLLs as `binaries`.

Required DLLs:

- `MSVCP140.dll`
- `VCRUNTIME140.dll`
- `VCRUNTIME140_1.dll`

Example `.spec` entry:

```python
import os
from PyInstaller.utils.hooks import collect_dynamic_libs

# Locate MSVC runtime DLLs from the Python environment
msvc_dlls = []
for dll in ["MSVCP140.dll", "VCRUNTIME140.dll", "VCRUNTIME140_1.dll"]:
    for search_path in [sys.base_prefix, r"C:\Windows\System32"]:
        full_path = os.path.join(search_path, dll)
        if os.path.exists(full_path):
            msvc_dlls.append((full_path, "."))
            break

a = Analysis(
    ...
    binaries=msvc_dlls,
    ...
)
```

The application MUST NOT assume these DLLs are present on the target machine.

# Required Project Structure

Projects MUST follow this structure:

```text
project/
├── backend/                  Python FastAPI app — business logic, API routes, database access
├── frontend/                 React/Vite source — developer only, compiled but never shipped as source
│   ├── package.json          Pinned frontend dependencies (no ^ or ~ version ranges)
│   ├── vite.config.js        Vite build configuration
│   └── tailwind.config.js    Tailwind CSS configuration
├── assets/                   Static files that ship with the app (icons, images, fonts)
├── data/                     Seed data and initial schema scripts used at first run
├── launcher/                 Main entry point — starts FastAPI, opens PyWebView, manages shutdown
├── packaging/                PyInstaller configuration
│   └── app.spec              Controls exactly what gets bundled (DLLs, assets, console=False)
├── dist/                     Final distributable output — produced by package.bat, this is what gets zipped
├── build.bat                 Developer script — installs dependencies and compiles the React frontend
├── package.bat               Developer script — runs build.bat then PyInstaller, produces dist/AppName/
├── requirements.txt          Pinned Python dependencies (exact == versions only)
└── README.md                 Developer documentation
```

Key distinctions:

- `frontend/` is a **build-time only** folder. Node.js and npm are required on the developer's machine to compile it, but the compiled static assets are what gets bundled — the source is never shipped to the end user.
- `data/` holds **seed and schema data** used at first run. It is NOT where runtime user data is stored. All user-generated data MUST go under `%LOCALAPPDATA%/CompanyApps/AppName`.
- `dist/` is the **only folder that matters to the end user**. Everything else is developer scaffolding. `package.bat` produces `dist/AppName/` which is zipped and distributed.

## WebView2 Preflight Check

PyWebView depends on the Microsoft Edge WebView2 runtime. On Windows 11 it is pre-installed. On Windows 10 it may be absent.

`Run_App.bat` MUST check for WebView2 before launching the executable.

If WebView2 is absent, the batch file MUST display a clear, user-friendly error message explaining what is missing and where to download it. It MUST NOT silently fail or produce a blank window.

Required preflight check logic:

```bat
reg query "HKLM\SOFTWARE\WOW6432Node\Microsoft\EdgeUpdate\Clients\{F3017226-FE2A-4295-8BDF-00C3A9A7E4C5}" >nul 2>&1
if %errorlevel% neq 0 (
    echo.
    echo  This application requires the Microsoft Edge WebView2 Runtime.
    echo  It does not appear to be installed on this machine.
    echo.
    echo  Please download and install it from:
    echo  https://developer.microsoft.com/en-us/microsoft-edge/webview2/
    echo.
    pause
    exit /b 1
)
```

This check MUST appear before the executable is launched.
